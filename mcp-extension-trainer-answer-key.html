1. Target Solution

The recommended reference solution is a customer product overview extension. It is the cleanest path through the lab because it reuses the existing customer and product data, adds a single clear endpoint, and produces a strong demonstration of why an MCP tool is useful.

1

New endpoint

GET /customers/{customer_id}/product-overview

2

New MCP tool

get_customer_product_overview(customer_id)

3

Supporting context

A guide resource or review prompt for interpreting the overview output

2. Expected Dataset

The minimal solution does not need a brand-new domain model. Trainers can accept a solution that reuses CUSTOMERS and PRODUCTS and enriches one with the other. If participants add a small guide dataset as well, that is ideal.

Suggested guide content in app/data.py

PRODUCT_OVERVIEW_GUIDE = """
Customer Product Overview Guide - Demo

1. Use product_id for integration logic.
2. Use name and description for summaries.
3. required_checks is intended for internal operational guidance.
4. Do not expose developer-only notes to end users.
5. Compare products by domain when summarizing portfolios.
""".strip()

3. Expected API Endpoint

A strong answer adds a new endpoint in app/internal_api.py that keeps the existing auth and error handling pattern. The endpoint should aggregate product details for the customer's active products. 

@app.get("/customers/{customer_id}/product-overview")
def get_customer_product_overview(customer_id: str, x_api_key: str | None = Header(default=None)):
    require_demo_api_key(x_api_key)
    customer = CUSTOMERS.get(customer_id)
    if not customer:
        raise HTTPException(status_code=404, detail="Customer not found")

    products = []
    for product_id in customer["active_products"]:
        product = PRODUCTS.get(product_id)
        if product:
            products.append(product)

    return {
        "customer_id": customer["customer_id"],
        "segment": customer["segment"],
        "risk_profile": customer["risk_profile"],
        "products": products,
    }
Acceptable variants: participants may omit risk_profile, rename the route, or include only selected product fields. The key requirement is that the endpoint provides a clear, enriched response instead of just forwarding the existing customer object.

4. Expected MCP Additions

Expected tool

@mcp.tool
def get_customer_product_overview(customer_id: str) -> dict[str, Any]:
    """Retrieve enriched product data for a customer."""
    return internal_get(f"/customers/{customer_id}/product-overview")

Expected resource

@mcp.resource("guide://customer-product-overview")
def customer_product_overview_guide() -> str:
    return PRODUCT_OVERVIEW_GUIDE

Optional strong answer: prompt template

@mcp.prompt
def customer_product_review_prompt(customer_id: str) -> str:
    return f"""
Review the product overview for customer {customer_id}.

Use the guide://customer-product-overview resource.
Return:
1. Product summary
2. Products with the highest control burden
3. Missing information
4. Recommended next actions
""".strip()

5. Validation Steps

Use these checks to confirm that the participant's solution works and that the extension is end-to-end complete.

  • Run uv run bank-api and verify the server starts successfully.
  • Call the new endpoint directly with the demo API key.
  • Run uv run bank-mcp and confirm the MCP server starts.
  • Use Copilot or another MCP client to call the new tool.
  • If a resource or prompt was added, verify that the client can use it as part of a final answer.

Sample direct API call

curl -H "x-api-key: training-demo-key" http://127.0.0.1:8000/customers/CUST-1001/product-overview

6. Demo Flow

1
Show the gap

Explain that the starter server can fetch customer and product data separately, but not as one enriched answer.

2
Show the new endpoint

Open the FastAPI docs or code and show the new product overview route.

3
Show the MCP tool

Open app/mcp_server.py and show the new tool wrapping the route.

4
Run the prompt

Ask Copilot to retrieve the product overview for CUST-1001 and summarize the products by domain.

Suggested demo prompt

Use the internal MCP server to retrieve the product overview for customer CUST-1001 and summarize which product has the highest operational control burden.

7. Assessment Guidance

Area What good looks like
Scope control The participant implements one coherent scenario instead of many disconnected changes.
Layering The API owns aggregation logic and the MCP layer remains a thin, clear wrapper.
Naming Tool and resource names are explicit and understandable to other developers.
Validation The solution is demonstrated through direct API calls and through an MCP prompt.
Enterprise thinking The participant preserves auth checks, avoids unnecessary data exposure and explains why MCP is the right layer.

2. What is MCP?

MCP stands for Model Context Protocol. It is a standard way for AI applications to discover and use external tools, data sources and services. Instead of letting an AI assistant guess how a system works, an MCP server exposes approved capabilities in a structured way.

🤖

MCP Client

The AI application, IDE, agent or assistant that wants to use external capabilities.

🟧

MCP Server

The secure intermediary that exposes tools, validates requests and formats responses.

🗄

Tools & Data

The systems behind the server, such as documentation, databases, GitHub, APIs or internal platforms.

Simple explanation

Think of an MCP server as a controlled toolbox for AI. The AI assistant does not directly access everything. Instead, it asks the MCP server what tools are available and calls those tools with clear parameters.

Trainer note: A useful analogy is a hotel concierge. The guest does not walk into every back-office system. The concierge knows what services are available, checks what is allowed, performs the request and returns the result.

3. MCP Server Architecture

An MCP architecture usually contains three main parts: the client, the MCP server and the connected tools or data sources.

🤖
MCP Client GitHub Copilot, IDE, AI app or agent
→
🟧
MCP Server Tool discovery, validation, execution and response formatting
→
🧰
Tools & Data Sources Documentation, RDS packages, APIs, repositories, databases or internal services

Core responsibilities

🔎

Tool discovery

Expose what the client can ask the server to do.

🛡

Access control

Validate access, permissions and allowed actions.

⚙️

Tool execution

Run the selected capability with the provided parameters.

📦

Response formatting

Return structured results the client can use reliably.

4. Rabobank Design System MCP Server

The RDS MCP Server helps developers build user interfaces that follow the Rabobank Design System. It connects GitHub Copilot or another MCP client to approved RDS components, documentation, examples and best practices.

What it helps with

  • Finding the right RDS component
  • Understanding component properties and variations
  • Generating RDS-compliant UI examples
  • Following accessibility and design guidelines
  • Reducing inconsistent or generic UI code

Example developer prompt

Prompt:

Create a customer registration form using Rabobank Design System components. Include first name, last name, email and a primary submit button.

<rds-form>
  <rds-text-field label="First name" required></rds-text-field>
  <rds-text-field label="Last name" required></rds-text-field>
  <rds-text-field label="Email" type="email" required></rds-text-field>
  <rds-button variant="primary">Register</rds-button>
</rds-form>

Possible exposed capabilities

Capability What the developer can ask Example result
Search components Which RDS component should I use for a primary action? Recommended component: rds-button, variant primary.
Get documentation What properties are available for the RDS button? Properties, variants, accessibility guidance and usage notes.
Generate example Create a login form with RDS components. RDS-compliant markup with form fields and button usage.
Check best practices Is this form accessible and aligned with the design system? Feedback on labels, validation, keyboard usage and consistency.

5. Access & Setup

Before using the RDS MCP Server, make sure you have the required access, tools and local setup.

Access checklist

  • Have a GitHub Copilot account.
  • Request access to the required AAD group.
  • Register to GitHub using your Rabobank email.
  • Link your GitHub account to the Rabobank organization via SSO.
  • Wait for the access sync to complete.
  • Install VS Code version 1.116 or later.
  • Install the GitHub Copilot extension and sign in.

Local RDS package setup

Create or update an .npmrc file in the project root:

registry=https://nexus3.rabobank.nl/repository/npmjs/

Install the required packages:

npm install @rabobank/rds-components@5.0.0
npm install @rabobank/rds-style@5.0.0

Minimal HTML template

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>My RDS App</title>

  <script type="module"
    src="./node_modules/@rabobank/rds-components/dist/rds-components/rds-components.esm.js">
  </script>

  <link rel="stylesheet" href="./node_modules/@rabobank/rds-style/css/rds-style.css" />
  <link rel="stylesheet" href="./node_modules/@rabobank/rds-style/fonts/myriad/default.css" />
</head>
<body>
  <rds-button>Click Me</rds-button>
</body>
</html>

Run locally:

npx http-server

7. Using the MCP Server Effectively

A good MCP prompt is specific about the desired outcome, the target component, constraints and the quality requirements.

Good prompt pattern

Create a [UI element or page]
using Rabobank Design System components.

Requirements:
- [field/component 1]
- [field/component 2]
- [validation rule]
- [accessibility requirement]
- [layout requirement]

Return:
- RDS-compliant code
- short explanation
- any assumptions

Example prompts

  • Create an accessible login form using RDS components.
  • Show me the correct RDS component for a destructive action.
  • Generate a customer overview card with status, balance and action button.
  • Review this HTML and suggest RDS-compliant improvements.
  • Create a form with validation messages and accessible labels.

Prompt quality checklist

  • Tell Copilot that it must use Rabobank Design System components.
  • Describe the business context, not only the technical component.
  • Ask for accessibility and validation guidance.
  • Ask for assumptions when information is missing.
  • Review generated code against official project standards before committing.

8. Guided Labs

Complete these labs in order. Each lab builds confidence with MCP-assisted development and prepares you for the final custom MCP-server assignment.

Lab 1 — Explore the RDS MCP Server

Goal: Understand what the server can help you with.

  1. Open VS Code with GitHub Copilot enabled.
  2. Ask Copilot what RDS components are available for forms.
  3. Ask for a simple RDS button example.
  4. Ask for accessibility best practices for forms.

Deliverable: A short note with three useful things you discovered.

Lab 2 — Build a simple RDS page

Goal: Generate and run a local RDS page.

  1. Create a local HTML file.
  2. Install the RDS packages.
  3. Ask Copilot to generate a profile form using RDS components.
  4. Run the page locally with npx http-server.

Deliverable: A working local page with at least three RDS components.

Lab 3 — Improve generated UI

Goal: Use MCP guidance to improve quality.

  1. Ask Copilot to review your RDS page.
  2. Focus on accessibility, labels, validation and layout.
  3. Apply at least three improvements.
  4. Write down what changed and why.

Deliverable: Before/after notes with improvements.

Lab 4 — Design an MCP use case

Goal: Prepare for building your own MCP server.

  1. Choose a developer workflow that could benefit from MCP.
  2. Define at least two tools your MCP server should expose.
  3. Define the input and output for each tool.
  4. Describe security and validation considerations.

Deliverable: A one-page MCP server design proposal.

9. Final Assignment — Build Your Own MCP Server

Build a small MCP server that exposes useful tools to an AI client. The goal is not to create a large production system, but to understand how MCP server design works in practice.

Choose one scenario

📚

Documentation MCP Server

Expose internal documentation or Markdown files as searchable knowledge tools.

  • search_docs
  • get_doc_section
  • summarize_doc
🧪

Testing MCP Server

Expose tools for checking local HTML pages, test files or generated UI output.

  • list_test_files
  • run_basic_check
  • generate_test_plan
🧩

Component Helper MCP Server

Expose a small component catalog with examples and usage rules.

  • search_components
  • get_component_example
  • validate_component_usage

Minimum requirements

  • Your MCP server exposes at least two tools.
  • Each tool has a clear name, description, input schema and output.
  • The server validates basic input before executing a tool.
  • The server returns structured, predictable results.
  • You include a README with setup instructions and sample prompts.

Suggested implementation approach

1
Design

Choose the use case and define the tools.

2
Implement

Create the MCP server and tool handlers.

3
Test

Call each tool with valid and invalid input.

4
Connect

Use the server from an MCP-compatible client.

5
Demo

Show a useful end-to-end workflow.

Example project structure

my-mcp-server/
├── src/
│   ├── server.js
│   ├── tools/
│   │   ├── searchDocs.js
│   │   └── getComponentExample.js
│   └── data/
│       └── components.json
├── package.json
└── README.md

Example README sections

  • Purpose of the MCP server
  • Available tools
  • Input and output examples
  • Setup instructions
  • Sample prompts
  • Known limitations

10. Assessment Rubric

Conceptual understanding The participant can explain what an MCP server does and how it differs from a regular API or backend service.
Tool design The server exposes useful, clearly named tools with understandable input and output.
Implementation quality The server runs locally, handles valid requests and deals with basic invalid input gracefully.
Developer usefulness The MCP server solves a realistic developer problem and improves an AI-assisted workflow.
Security awareness The participant can explain what access control, validation or guardrails would be needed in an enterprise environment.
Documentation The README explains setup, tools, sample prompts and limitations clearly.

11. MCP Cheat Sheet

MCP Server

A server that exposes tools and resources to AI clients in a standardized way.

Tool

A callable capability, such as search documentation, get component details or run a check.

Resource

Content or context that the client can read, such as files, docs, schemas or catalog data.

Good MCP use cases

  • Design systems
  • Internal documentation
  • Developer tooling
  • Testing helpers
  • Knowledge bases

Avoid

  • Unrestricted data access
  • Unclear tool names
  • Weak input validation
  • Returning unstructured output
  • Tools without a real use case

Strong sample prompt

Use the MCP server to find the correct component for a customer contact form. Generate RDS-compliant code and explain the accessibility considerations.

Final reflection: What would make your MCP server safe, useful and maintainable in a large enterprise environment?

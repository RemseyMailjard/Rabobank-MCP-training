1. Overview

This lab is part of the Rabobank MCP training. Participants start from the current demo project and extend it with additional capabilities. The focus is not only on adding code, but on making clear design choices about when something should be exposed as a tool, a resource or a prompt.

🎯

Main goal

Extend the MCP server so Copilot can do more than in the starter version.

🔌

API work

Add at least one new internal endpoint that models a useful business capability.

🟧

MCP work

Expose the new capability as an MCP tool and add supporting context through a resource or prompt.

✅

Validation

Prove the extension works through API calls and Copilot prompts.

2. Learning Goal

After this lab, a developer should understand how to evolve the current MCP server in a controlled, enterprise-friendly way. That means adding business value without breaking the structure of the existing demo.

Expected learning outcomes

  • Explain when a capability should become an MCP tool, resource or prompt.
  • Add new demo data and expose it through the internal FastAPI layer.
  • Create at least one new MCP tool that calls the new API endpoint.
  • Add a resource or prompt that improves how developers use the new capability.
  • Test the extension end-to-end with uv, the API docs and Copilot prompts.
Training intent: the strongest submissions are not the biggest ones. The best result is a small, coherent extension that clearly demonstrates why the MCP layer exists.

5. Core Scenarios

Choose at least one of the first three scenarios. The fourth is a stretch goal.

Scenario A: Customer product overview

Add a new endpoint that returns the active products of a customer enriched with product details. Then expose it through an MCP tool such as get_customer_product_overview.

Scenario B: Controlled action

Add a POST endpoint that performs a review or follow-up action, then expose it as an MCP tool. This demonstrates the difference between read-only data and controlled actions.

Scenario C: New policy or guide

Add a new resource that explains how developers should interpret the new endpoint or tool output. This shows how resources complement tools.

Scenario D: Reusable review prompt

Add an MCP prompt that guides Copilot through a structured review of your new capability. This creates consistent AI output across users.

Recommended minimum scope

  • One new internal API endpoint.
  • One new MCP tool that calls it.
  • One new resource or prompt that supports the same scenario.
  • Two demo prompts that prove the extension works.

6. Implementation Guide

The most practical path for this project is to add a customer product overview scenario. It reuses existing data, requires little code and gives a very clear improvement over the current demo.

Suggested endpoint

Add a new route in app/internal_api.py:

@app.get("/customers/{customer_id}/product-overview")
def get_customer_product_overview(customer_id: str, x_api_key: str | None = Header(default=None)):
    require_demo_api_key(x_api_key)
    customer = CUSTOMERS.get(customer_id)
    if not customer:
        raise HTTPException(status_code=404, detail="Customer not found")

    products = [PRODUCTS[product_id] for product_id in customer["active_products"]]
    return {
        "customer_id": customer_id,
        "segment": customer["segment"],
        "products": products,
    }

Suggested tool

Add a matching MCP tool in app/mcp_server.py:

@mcp.tool
def get_customer_product_overview(customer_id: str) -> dict[str, Any]:
    """Retrieve enriched product data for a customer."""
    return internal_get(f"/customers/{customer_id}/product-overview")

Suggested supporting resource

PRODUCT_OVERVIEW_GUIDE = """
Customer Product Overview Guide - Demo

1. Use product_id for integration logic.
2. Use name and description for summaries.
3. required_checks is intended for internal operational guidance.
4. Do not expose developer-only notes to end users.
5. Compare products by domain when summarizing portfolios.
""".strip()
Design note: do not add random extra features. Keep the extension coherent. A small, complete scenario teaches more than five unrelated additions.

Clear scenario The extension solves one coherent developer or business question instead of many unrelated ones.
Correct layering The internal API owns the data and logic, while the MCP layer exposes it cleanly to clients.

1. Overview

This lab is built around the existing internal banking demo MCP server in this repository. Participants extend the current solution instead of starting from scratch. That means they will work across the existing FastAPI layer, the MCP layer and the training prompts that expose the new capabilities to GitHub Copilot.

🎯

Goal

Learn how to add new capabilities to an MCP server in a structured, realistic way.

🧱

Starting point

Use the current demo with customer, product, API catalog, policy and architecture tools.

🟧

Focus

Extend internal APIs first, then expose the capability through MCP.

✅

Result

Copilot can use newly added tools, resources and prompts during the demo.

2. Learning Goal

By the end of the lab, a developer should be able to explain where new functionality belongs in the project, implement it in the internal API, expose it via MCP, and validate the result with practical prompts and test calls.

Expected learning outcomes

  • Explain when a capability should be implemented as an MCP tool, resource or prompt.
  • Add new demo data and new endpoints without breaking the existing structure.
  • Create at least one new MCP tool that calls a new internal API endpoint.
  • Create at least one new MCP resource or reusable prompt.
  • Demonstrate the extension with working API calls and Copilot prompts.
Trainer note: The point of the lab is not only to add code. The deeper learning goal is to show how an MCP server becomes a controlled AI-facing layer over internal systems.

3. Extension Architecture

Participants should work from the inside out. Add or adapt demo data, expose it through the internal API, then wire it into MCP as a tool, resource or prompt.

4. Lab Flow

Study the current server Review the existing customer, product, API catalog, policy and architecture capabilities. Identify where data lives, how the API exposes it and how MCP wraps it.
Add a new internal capability Create a realistic new scenario such as customer product overview, service signals, onboarding follow-up or review guidance.
Create the API endpoint Build a GET endpoint for read-only data and optionally a POST endpoint for a controlled action or review flow.
Expose it through MCP Add a tool for targeted use, a resource for stable knowledge, and optionally a prompt that standardizes the analysis output.
Validate and demo Test both the API and the MCP layer. The lab is only complete if Copilot can use the newly added capability in a clear business scenario.

5. Lab Scenarios

Choose the first three scenarios as the core lab. Use the fourth as a stretch goal.

Scenario A - Customer Product Overview

Business need: A developer wants one MCP call that returns a customer and enriched product details together.

Goal: Add a new read endpoint such as /customers/{customer_id}/product-overview and expose it as an MCP tool.

Scenario B - Controlled Follow-Up Action

Business need: Demonstrate a safe write-like action for training without touching real systems.

Goal: Add a POST endpoint such as /onboarding/cases/{caseId}/follow-up and expose it as a new MCP tool.

Scenario C - Developer Guidance Resource

Business need: Teams want consistent explanation of newly added fields, checks and response meanings.

Goal: Add a new resource such as guide://customer-product-overview or policy://data-minimization.

Scenario D - Review Prompt Template

Business need: Free-form prompts produce inconsistent reviews.

Goal: Add a prompt template that returns a structured review with strengths, missing controls, questions and next actions.

6. Development Tasks

Task 1 - Add new demo data

Extend app/data.py with a new small, explainable dataset. Keep it fake, readable and clearly related to the business scenario you picked.

CUSTOMER_SIGNALS = {
    "CUST-1001": {
        "status": "stable",
        "signals": ["salary-inflow-consistent", "mortgage-payments-on-time"]
    },
    "CUST-2002": {
        "status": "review",
        "signals": ["seasonal-cashflow", "manual-document-follow-up"]
    }
}

Task 2 - Add an API endpoint

Create a new internal API endpoint in app/internal_api.py that exposes the new scenario with the same demo auth pattern and error handling as the rest of the file.

@app.get("/customers/{customer_id}/signals")
def get_customer_signals(customer_id: str, x_api_key: str | None = Header(default=None)):
    require_demo_api_key(x_api_key)
    signals = CUSTOMER_SIGNALS.get(customer_id)
    if not signals:
        raise HTTPException(status_code=404, detail="Signals not found")
    return {"customer_id": customer_id, **signals}

Task 3 - Add a new MCP tool

Translate the new endpoint into a clear MCP capability in app/mcp_server.py.

@mcp.tool
def get_customer_signals(customer_id: str) -> dict[str, Any]:
    """Retrieve support signals for a customer."""
    return internal_get(f"/customers/{customer_id}/signals")

Task 4 - Add a resource or prompt

Create stable supporting context for developers and AI. This turns the extension into a fuller MCP experience rather than a single tool call.

@mcp.resource("guide://customer-signals")
def customer_signals_guide() -> str:
    result = internal_get("/policies/customer-signals-guide")
    return result["content"]
